# CLAUDE.md — 全局通用配置

## 通用原则

- 代码质量：函数 < 50 行，文件 < 800 行
- 结构化日志，禁止裸 console.log / print
- 输入校验 + 错误处理 + 统一错误响应
- 不暴露 token/密码/密钥到日志或前端

## Superpowers 插件已安装

本项目已启用超级自动化流水线。**请彻底放弃手动分阶段调用（如单独调用 brainstorming）。**

## 全自动流水线触发规则（最高优先级）

当用户在会话中输入任何“业务需求、功能需求、或题目描述”时，**必须直接进入全自动流程，绝对不允许在某个环节（环境）完成后停下不继续**：

1. **绝不翻译、绝不自行扩写**：将用户给到的完整需求提示词**原文（字对字，纯复制）**写入当前目录的 `prompt.md` 中。如果用户输入的是英文，`prompt.md` 中绝不可出现哪怕一个中文字符！
2. 自动生成 `TASK-ID`（格式：`TASK-YYYYMMDD-HHMMSS`）。
3. **立即一键执行全自动子代理流水线（具备多阶段自研能力）**：
   ```bash
   /superpowers:prompt2repo "请严格基于 prompt.md 中的业务需求进行架构设计与全流程开发" --task-id <TASK-ID>
   ```

**执行约束与防暴走纪律：**

- **严禁**擅自缩水流程。必须通过 prompt2repo 流水线，依次调用内置的子代理完成解析、计划、执行、自测和打包全过程！
- **禁止**弹出“A/B/C 选择菜单”阻塞用户，自动分析并选择最佳推荐项继续！
- 🔴 **绝对拦截，禁止单次回合越权 (CRITICAL)**：“全自动”是指依赖外部流水线系统进行状态流转，**绝不允许 AI 在单次对话回复中一口气写完所有代码**。在你单次的上下文回复中：
  1. 必须且只能执行当前所处的唯一 Phase 任务。
  2. 绝不允许在正式进入 Phase 2 (Execution) 前，使用 `mkdir`、`Write` 等工具直接创建业务代码文件。所有流程必须先沉淀为设计文档。
- 必须遵循 `AGENTS.md` / `CLAUDE.md` 的规范优先级。
- 不得使用 mock 数据替代真实逻辑。
- 关键结论需给出可追溯证据。

## 题目交付准入与一票否决规则 (CRITICAL)

**在执行需求时，必须严格遵守以下红线，任何违规将导致交付物不可验收（一票否决）：**

1. **硬性门槛（绝对可跑，环境隔离）：**
   - 必须支持 `docker compose up` 一键无障碍启动。
   - 严禁代码依赖你的本地绝对路径、全局环境变量或宿主机未声明的服务。
   - README文档必须真实有效。
2. **严格扣题（拒绝擅自简化与 Mock）：**
   - 必须严格围绕 Prompt 的核心业务目标开发，严禁删减核心需求或擅自简化技术实现（如将 WebSocket 简化为轮询）。
   - 除非明确要求，核心业务逻辑必须真实实现，严禁硬编码（如写死的返回值）。
3. **交付完整性（0到1的工程）：**
   - 必须是具有清晰目录结构（如 src, config 等）的完整工程，提供完整的配置/依赖文件。
   - 严禁单文件几千行代码，拒绝交付局部片段。
4. **代码整洁度与架构质量：**
   - 架构分层必须合理（如前后端分离，路由/业务/数据隔离，组件合理拆分）。
   - 清理所有冗余目录（无 `node_modules/`, `.venv/` 等），去除个人密钥敏感信息。
   - 删除废弃代码和无意义的调试语句（如 `print()`, `console.log()`）。
   - API 接口返回值必须结构清晰、格式化良好。
5. **专业度（测试、错误处理与日志）：**
   - 必须提供完整可执行的测试方案，包含 `unit_tests/` 和 `API_tests/`，并提供能一键执行的 Shell 脚本。
   - 健壮的错误处理：提供标准的 HTTP 状态码和清晰的 JSON 错误提示，严禁直接抛出堆栈导致崩溃；前端必须有容错提示。
   - 关键业务流程必须有上下文清晰的结构化日志记录。
   - 必须执行输入防御与基本安全。
